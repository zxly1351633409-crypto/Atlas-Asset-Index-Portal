# Windows 便携演示与跨电脑连接

## 1. 交付物说明

`Atlas-Portal-Windows-v0.2.0.zip` 是联网引导包。它包含程序源码、锁定依赖清单、连接器、启动脚本和使用说明，不包含本机资源、账号配置、`node_modules` 或 Node.js 二进制。

首次启动会自动完成环境检查；后续启动复用 `.runtime`，不会重复下载和构建。

| 文件 | 用途 |
| --- | --- |
| `连接资源目录.bat` | 首次选择本机文件夹或 UNC 路径，也可重新配置 |
| `启动 Atlas.bat` | 检查环境、补齐依赖、启动监视器和网页并打开浏览器 |
| `打开 Atlas.bat` | Atlas 已运行时重新打开页面；未运行时会自动启动 |
| `停止 Atlas.bat` | 结束由启动器创建的网页服务和目录监视器 |
| `首次使用说明.txt` | 适合随 ZIP 一起查看的最短说明 |

## 2. 单机演示

1. 将 ZIP 解压到用户有写权限的位置，例如桌面或 `D:\Tools\Atlas`。
2. 运行 `连接资源目录.bat`，直接回车可使用包内 `Atlas-Data`，也可输入其他目录。
3. 运行 `启动 Atlas.bat`，等待浏览器打开。
4. 把图片放入 `ProjectAurora\Scenes\J 剧情 模块\1.0`。
5. 在页面选择“场景”，进入 `J 剧情 模块` 和 `1.0`，双击图片查看预览。

## 3. 两台电脑模拟 NAS

### 资源提供电脑

1. 建立测试目录，例如 `D:\AtlasShare`。
2. 在文件夹属性的“共享”页中创建共享名，例如 `AtlasShare`。
3. 同时检查“共享权限”和 NTFS“安全”权限；测试账号至少需要读取权限，需要演示新增/修改时再给写入权限。
4. 记录电脑名，可在 PowerShell 运行 `$env:COMPUTERNAME` 查看。
5. 得到 UNC 路径：`\\电脑名\AtlasShare`。

### 演示电脑

1. 确保两台电脑处于可互访网络，必要时连接公司 VPN。
2. 在资源管理器地址栏先输入 UNC，完成 Windows 凭证验证。
3. 解压 Atlas，运行 `连接资源目录.bat`，输入同一 UNC。
4. 运行 `启动 Atlas.bat`。
5. 在资源提供电脑新增图片；监视器更新索引后，演示页面会读取最新清单。

本机已通过 Windows UNC 共享完成验证：扫描结果与本机路径一致，识别出 2 个模块、5 条内容和 5 个文件。

## 4. 首次启动的环境引导

```text
读取 config/local.user.json
        ↓
检查资源目录是否可访问
        ↓
检查 Node.js 22.13+
        ↓ 不存在或版本过低
下载 Node 官方 latest-v22.x Windows ZIP
        ↓
核对 SHASUMS256.txt
        ↓
npm ci 安装 package-lock.json 锁定依赖
        ↓
构建、扫描、启动监视器和网页
        ↓
健康检查通过后打开浏览器
```

运行环境保存在 `.runtime`，本机路径保存在 `config/local.user.json`；两者均被 Git 忽略。

## 5. 联网包与离线包

| 类型 | 优点 | 限制 |
| --- | --- | --- |
| 联网引导包 | ZIP 小，适合 GitHub；自动获取当前 Node 22 安全修订和锁定依赖 | 首次运行依赖 Node.js/npm 网络或公司代理 |
| 完全离线包 | 现场无需互联网，可直接启动 | 体积大，需按 Windows 架构制作，并定期更新依赖和安全修订 |

本次交付的是联网引导包。若公司演示网络受限，建议由技术部门在公司网络内执行一次启动，再将已经形成的 `.runtime`、`node_modules` 和构建产物制作成内部离线包；不建议把大型离线包直接提交到 Git 历史。

## 6. 公司真实目录的边界

标准演示结构可自动识别，但公司历史资源不一定使用相同层级。直接把任意旧目录连上并不代表门户能可靠判断工作域、模块和项目版本。

正式试点建议增加：

- 原路径来源视图，完整保留 TFVC/NAS 目录树。
- 映射表：`原路径 → 工作域 / 模块 / 项目版本`。
- 待分类队列，未知内容不自动乱分。
- TFVC changeset/label 与业务版本的映射。
- 只读服务账号、最小权限、下载审计和缓存保留策略。

## 7. 常见故障

| 现象 | 检查方式 |
| --- | --- |
| 启动窗口提示目录不可访问 | 先在资源管理器打开本机/UNC 路径，确认账号和网络 |
| 页面显示“正在连接资源目录”后报错 | 查看 `.runtime\logs\watcher.err.log`，重新运行连接脚本 |
| 3011 已被占用 | 启动器会尝试后续 20 个端口，并显示实际页面地址 |
| 首次依赖下载很慢 | 检查代理、npm 网络；缓存成功后不会重复下载 |
| 新文件没有出现 | 文件名不要以下划线开头；检查是否位于工作域/模块/版本下 |
| 双击 BAT 一闪而过 | 当前版本会暂停显示结果；确认使用完整 v0.2.0 包而非旧 BAT |

## 8. GitHub 与技术交接

- GitHub 仓库保留源码、`package-lock.json`、配置示例、测试、文档和打包脚本。
- 本机配置、缓存、真实连接器清单和真实资源不提交。
- 仓库提供 `docs/verify-and-package.workflow.example.yml` 模板；技术部门用具备 `workflow` 权限的账号复制到 `.github/workflows/` 后，GitHub Actions 会在 Windows 上执行测试并生成 ZIP Artifact。
- 技术部门接手时优先评审连接器边界、账号权限、增量索引和缓存，而不是先重做页面。
- 公开仓库前检查截图中的电脑名、路径和演示图片版权；不确定时使用私有仓库。
